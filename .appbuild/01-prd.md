# 01-PRD — 365일 말씀 묵상 시집 (공개 웹사이트)

## 목적 (Why)
현재 구글드라이브 폴더에 파일로만 쌓이고 있는 "365일 말씀 묵상 시집" 원고를, 매일 한 편씩
읽을 수 있는 공개 웹사이트로 대체한다. 콘텐츠(성경 구절 + 묵상시)가 주인공인 "시집" 형태의
읽기 경험을 제공하는 것이 목적이며, 관리자용 CMS나 편집 기능은 이번 범위에 포함하지 않는다
(콘텐츠 원본은 `../집필부/day_XXX.md`에서 계속 작성되고, 이 사이트는 그 결과물을 정적으로
렌더링만 한다).

## 타깃 사용자
- **독자**: 시집을 처음부터 읽거나, 특정 월/날짜를 찾아 읽는 일반 방문자.
- **오너/집필부/감수부**: 목차에서 전체 진행 현황(365일 중 몇 편 공개됐는지)을 한눈에 확인.

## 범위 (이번 1차 슬라이스)
appbuild 04-tasks 1차 슬라이스 = **표지 + 목차 + day_001~003 상세 페이지**가 실제로
브라우저에서 보이고 서로 링크로 연결되어 데모 가능한 상태.

### 기능 목록 (수용 기준 포함)

**F1. 표지 페이지**
- 시집 제목("365일 말씀 묵상 시집")과 짧은 부제(마스터플랜 개요 1~2문장)가 보인다.
- 목차로 이동하는 명확한 진입점(링크/버튼)이 있다.
- 수용 기준: `/` 접속 시 제목·부제·목차 진입 링크가 렌더링된다.

**F2. 목차 페이지 (12개월 대주제 구조)**
- `master_plan.md`의 12개월 대주제 표를 그대로 목차 구조로 사용한다(1월 시작~12월 부활).
- 각 월 아래 해당 월에 속하는 day가 나열된다. day_001~003처럼 실제로 존재하는 글은
  링크로 클릭 가능해야 하고, 아직 쓰이지 않은 day(004~365)는 비활성(회색, 클릭 불가) 표시로
  "미공개"임이 시각적으로 구분되어야 한다.
- 수용 기준: `/toc` 접속 시 12개월 섹션이 모두 보이고, day_001·002·003 3개만 활성 링크이며
  나머지는 비활성 상태로 렌더링된다.

**F3. 일일 묵상 페이지**
- 원본 4요소를 그대로 유지해 렌더링한다: ①제목 ②성경 구절(출처 포함) ③묵상시 본문
  ④짧은 여운(산문).
- 수용 기준: `/day/1`, `/day/2`, `/day/3` 각각에서 day_001~003.md의 제목·성경구절·시·
  짧은 여운 텍스트가 원본과 누락 없이 렌더링된다(자간·줄바꿈 등 표현상 재구성은 허용되나
  텍스트 내용 손실·왜곡은 금지).

**F4. 이전/다음 날 네비게이션**
- 일일 페이지 하단에 "이전 날"/"다음 날" 링크를 둔다.
- 경계 처리: day_001에는 "이전 날" 링크가 없다(첫 편). 마지막으로 공개된 day(현재 day_003)에는
  "다음 날" 링크가 없다.
- 수용 기준: day_002 페이지에는 이전(day_001)·다음(day_003) 링크가 모두 보이고, day_001에는
  다음 링크만, day_003에는 이전 링크만 보인다.

**F5. 콘텐츠 마이그레이션**
- `../집필부/day_001.md` ~ `day_003.md`를 Astro content collection 형식으로 이관한다.
- 수용 기준: 3개 파일 모두 content collection에서 파싱 성공(빌드 에러 없음), 원문 4요소가
  프론트매터/본문에 1:1로 대응된다.

## 비범위 (이번 1차 슬라이스에서 제외)
- 검색, 구독/알림(RSS 등), 댓글/방명록, 로그인/인증 — 전부 없음(공개 정적 사이트).
- day_004 이후 콘텐츠 작성 — 집필부 소관, 이번 슬라이스는 마이그레이션 파이프라인만 검증.
- GitHub 원격 저장소 연결·push, Vercel 실배포 — 오너 승인이 필요한 "외부 발행"이므로 이번
  위임 범위는 **로컬 커밋까지**. Vercel 로그인 등 사람 개입이 필요한 지점은 도달 시 중단하고
  master에게 보고한다.
- 기존 instructor-site와의 코드/설정 공유 — 완전히 독립된 별도 프로젝트로 신규 구축.

## 비기능 요구
- **정적 사이트**: 서버리스, 전부 빌드타임 렌더링(Astro `output: 'static'`). 배포 시 서버
  프로세스 불필요.
- **읽기 경험 우선**: 시가 주인공 — 타이포그래피가 본문 가독성을 최우선으로 한다(디자인
  상세는 03-architecture·02-screens에서 구체화). 측정 가능 기준(형용사 "빠름" 대신 수치/이진
  조건): 빌드 산출물이 **100% 정적 HTML**이고 런타임 서버 호출·클라이언트 fetch가 **0건**
  이어야 한다(Astro `output: 'static'` 빌드 결과물 검사로 확인 가능).
- **반응형**: 모바일에서도 시 본문 줄바꿈·여백이 무너지지 않아야 한다.
- **콘텐츠 무결성**: 마이그레이션 과정에서 원문 텍스트가 잘리거나 오탈자가 생기지 않는다
  (감독관 검증 항목에 원문 대조 포함).
